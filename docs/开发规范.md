# 企业内部员工助手智能体 - 开发规范

---

## 1. 开发流程规范

### 1.1 小步开发原则
- **每次只开发一个小功能**
- 开发完成后，跑通测试
- 确认没问题后，git提交
- 再进行下一个功能

### 1.2 开发顺序
1. 先开发Java服务基础框架
2. 再开发Python Agent服务基础框架
3. 然后开发三条核心链路（按顺序：政策问答 → 资源查询 → 审批流）
4. 最后开发前端

## 2. 代码注释规范

总规范：
- 类上和方法上的注释才能使用/**/
- 其他地方比如字段注释、业务层代码注释一律使用//，不使用/**/

### 2.1 Java代码注释

#### Controller层
- 每个Controller类上添加类注释，说明该Controller的职责
- 每个接口方法上添加注释，说明：
  - 接口功能
  - 参数说明
  - 返回值说明

**示例**:
```java
/**
 * 对话接口Controller
 * 提供前端对话相关的API
 */
@RestController
@RequestMapping("/api/chat")
public class ChatController {

    /**
     * 发送对话消息
     *
     * @param request 对话请求，包含用户ID、消息内容、会话ID
     * @return 对话响应，包含助手回复、会话ID、附件信息
     */
    @PostMapping
    public ChatResponse chat(@RequestBody ChatRequest request) {
        // ...
    }
}
```

#### Service层
- 每个Service类上添加类注释
- 每个方法上添加注释，说明方法功能
- **每一大步逻辑添加行内注释**，但不要每行都注释

**示例**:
```java
/**
 * 工单服务
 * 提供工单创建、查询、审批等功能
 */
@Service
public class TicketService {

    /**
     * 创建工单
     */
    public Ticket createTicket(TicketCreateRequest request) {
        // 1. 生成工单号（TK + 年月日 + 4位序号）
        String ticketNo = generateTicketNo();

        // 2. 构建工单实体
        Ticket ticket = new Ticket();
        ticket.setTicketNo(ticketNo);
        ticket.setUserId(request.getUserId());
        ticket.setType(request.getType());
        ticket.setReason(request.getReason());
        ticket.setStatus(TicketStatus.PENDING);

        // 3. 保存到数据库
        ticketMapper.insert(ticket);

        return ticket;
    }
}
```

#### Mapper层
- 简单的CRUD方法不需要注释（MyBatis Plus自动生成）
- 自定义复杂查询方法需要添加注释

### 2.2 Python代码注释

#### API层 (FastAPI)
- 每个路由函数添加注释，说明接口功能

**示例**:
```python
@router.post("/chat")
async def chat(request: ChatRequest):
    """
    智能对话接口
    接收用户消息，调用LangGraph Agent处理，返回回复
    """
    # ...
```

#### Agent层 (LangGraph)
- 图定义添加注释
- 每个节点函数添加注释
- 关键逻辑步骤添加注释

**示例**:
```python
def create_agent_graph():
    """
    创建LangGraph智能体图
    节点顺序：意图识别 -> 任务拆解 -> 工具调用 -> 回答生成
    """
    graph = StateGraph(AgentState)

    # 添加节点
    graph.add_node("intent_recognition", intent_recognition_node)
    graph.add_node("tool_calling", tool_calling_node)
    graph.add_node("answer_generation", answer_generation_node)

    # 构建边
    graph.set_entry_point("intent_recognition")
    graph.add_edge("intent_recognition", "tool_calling")
    graph.add_edge("tool_calling", "answer_generation")
    graph.set_finish_point("answer_generation")

    return graph.compile()
```

#### Tools层
- 每个工具函数添加注释，说明功能、参数、返回值

## 3. Git提交规范

### 3.1 Commit Message格式
```
<type>(<scope>): <subject>

<body>
```

**Type类型**:
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 重构（既不是新增功能，也不是修改bug）
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**:
```
feat(Java): 搭建Spring Boot基础框架

- 初始化Spring Boot项目
- 配置MyBatis Plus
- 创建基础目录结构
```

### 3.2 提交时机
- 每个小功能完成并跑通后提交
- 文档更新后提交
- 不要攒一堆功能再提交

## 4. 代码结构约定

### 4.1 包名约定
- Java: 全小写，用点分隔，`com.example.xxx`
- Python: 全小写，用下划线分隔，`xxx_yyy`

### 4.2 类名约定
- Java: 大驼峰命名（PascalCase），`UserService`
- Python: 大驼峰命名（PascalCase），`AgentState`

### 4.3 方法/函数名约定
- Java: 小驼峰命名（camelCase），`getUserById`
- Python: 小写下划线分隔（snake_case），`get_user_by_id`

### 4.4 常量约定
- Java: 全大写下划线分隔，`MAX_RETRY_COUNT`
- Python: 全大写下划线分隔，`MAX_RETRY_COUNT`

## 5. 测试与模拟数据规范

### 5.1 模拟数据生成规范
- **位置**: Java的 `src/test/java/com/example/data/`
- **原则**: 一个方法只生成一个表的数据
- **命名**: `XxxDataGenerator`，`ResourceDataGenerator`

**示例**:
```java
/**
 * 资源模拟数据生成器
 */
public class ResourceDataGenerator {

    /**
     * 生成投影仪资源数据
     */
    public List<Resource> generateProjectors() {
        // ...
    }

    /**
     * 生成会议室资源数据
     */
    public List<Resource> generateMeetingRooms() {
        // ...
    }
}
```

### 5.2 测试类规范
- **位置**: Java的 `src/test/java/com/example/test/`
- **与数据生成类分开**
- 命名: `XxxServiceTest`

## 7. 其他约定

### 7.1 配置文件
- Java: `application.yml` 中数据库等配置用环境变量或默认值
- Python: `.env` 文件，不提交到Git（在.gitignore中），提供 `.env.example`

### 7.2 硬编码
- 避免硬编码魔法值
- 常量定义在常量类或配置文件中

### 7.3 日志
- 关键路径添加日志
- 异常堆栈要打印
