# 企业内部员工助手智能体 - 项目难点分析与解决方案

---

## 1. 概述

本文档分析项目落地过程中可能遇到的技术难点，并提供初步解决方案。作为简历项目，我们的目标是"最小可行演示"，在保证效果的前提下尽量降低实现复杂度。

---

## 2. 难点一：LangGraph智能体编排与状态管理

### 难点描述
- LangGraph是较新的技术，学习曲线较陡
- 需要设计合理的State状态结构
- 需要定义清晰的节点逻辑和边流转
- 多步骤任务拆解需要精确的Prompt设计
- 需要支持条件分支（不同意图走不同流程）
- 需要支持循环（工具调用后可能需要继续决策）

### 初步解决方案

#### 2.1 LangGraph完整设计（契合需求，不简化）
```
                    ┌───────────────┐
                    │   入口节点    │
                    │ (接收用户输入)│
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │  需求理解节点  │
                    │ (意图识别+任务拆解)│
                    └───────┬───────┘
                            │
            ┌───────────────┼───────────────┐
            │               │               │
    ┌───────▼───────┐      │      ┌────────▼──────┐
    │  政策问答分支  │      │      │  资源查询分支   │
    │ (RAG检索)    │      │      │ (查询资源)     │
    └───────┬───────┘      │      └────────┬──────┘
            │               │               │
            └───────────────┼───────────────┘
                            │
                    ┌───────▼───────┐
                    │  审批流分支    │
                    │ (创建工单)     │
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │  回答生成节点  │
                    │ (整合结果)     │
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │   结束节点     │
                    └───────────────┘
```

#### 2.2 State结构设计
```python
class AgentState(TypedDict):
    # 基础信息
    user_id: str
    session_id: str
    message: str

    # 需求理解结果
    intent: Optional[str]  # POLICY_QA, RESOURCE_QUERY, TICKET_CREATE, CHITCHAT
    tasks: List[Task]      # 拆解后的任务列表（支持多任务）

    # 工具调用相关
    current_task_index: int
    tool_calls: List[ToolCall]
    tool_results: List[ToolResult]

    # RAG相关
    retrieved_docs: List[Document]

    # 最终输出
    answer: str
    attachments: List[Attachment]
```

#### 2.3 分阶段实现（按功能迭代，不简化架构）
1. 第一阶段：搭建LangGraph基础框架（只有入口和结束节点）
2. 第二阶段：实现需求理解节点（意图识别+任务拆解）
3. 第三阶段：实现政策问答分支（RAG检索+回答生成）
4. 第四阶段：实现资源查询分支
5. 第五阶段：实现审批流分支（支持多工单创建）
6. 第六阶段：联调所有分支，完善条件流转逻辑

## 3. 难点二：RAG检索效果

### 难点描述
- 向量库检索的准确率取决于文档分块策略
- 需要调整检索Top-K和相似度阈值
- 没有真实政策文档，需要自己构造模拟数据

### 初步解决方案

#### 3.1 简化版RAG策略
- 文档格式：纯Markdown，用标题作为分块边界
- 分块大小：每个标题对应一个Chunk（不需要按Token切分）
- 向量模型：用Sentence-Transformers轻量模型（all-MiniLM-L6-v2）

#### 3.2 模拟政策文档准备
准备5个典型政策文档，每个文档3-5个章节：
1. `员工办公规范.md` - 宠物、着装、考勤
2. `IT设备管理办法.md` - 电脑申请、维修、归还
3. `会议室预订规范.md` - 预订流程、使用须知
4. `软件使用许可管理.md` - IDEA、Office等软件申请
5. `数据访问权限申请流程.md` - 数据库权限申请

#### 3.3 检索优化
- 初期用简单的关键词匹配 + 向量检索混合
- 如果效果不好，直接用"标题匹配"（把文档标题当关键词索引）

## 4. 难点三：Agent工具调用的准确性

### 难点描述
- 大模型可能会：
  - 调用错误的工具
  - 参数格式不对
  - 忘记调用工具直接回答
- 没有工具调用历史的Few-Shot示例

### 初步解决方案

#### 4.1 工具定义简化
每个工具只保留必要参数，参数类型用最简单的String：

```python
# 工具定义示例
def search_policy(keyword: str) -> List[dict]:
    """搜索政策文档"""
    pass

def query_resource(type: str) -> List[dict]:
    """查询资源，type: PROJECTOR/LAPTOP/ROOM/LICENSE"""
    pass

def create_ticket(type: str, reason: str) -> dict:
    """创建工单"""
    pass
```

#### 4.2 强制工具调用策略
- 在Prompt中明确：**必须先判断是否需要调用工具，再决定是否直接回答**
- 意图识别单独做一步，先分类，再根据意图决定调用哪个工具

#### 4.3 Fallback策略
- 如果工具调用失败/参数错误，捕获异常后让大模型重试一次
- 重试还不行，走降级方案：礼貌告知"暂时无法处理，请稍后再试"

## 5. 难点四：多任务拆解（审批流链路）

### 难点描述
用户的一个请求可能包含多个审批项：
> "帮我申请借用一台投影仪，下周二用，顺便帮我把公司业务数据库的只读权限开一下"

需要：
1. 识别出这是两个独立任务
2. 分别生成两个工单
3. 把两个工单结果合并回答

### 初步解决方案（完整实现，不简化）

#### 5.1 Task数据结构
```python
class Task(TypedDict):
    task_id: str
    task_type: str  # PROJECTOR_BORROW, DB_READ_PERMISSION, etc.
    reason: str
    metadata: Dict[str, Any]  # 存放时间、地点等扩展信息
    status: str  # PENDING, PROCESSING, COMPLETED, FAILED
```

#### 5.2 任务拆解Prompt设计
让大模型输出严格的JSON格式，用Few-Shot示例：
```python
TASK_SPLITTING_PROMPT = '''
你是一个任务拆解专家。请将用户的请求拆分为一个或多个独立任务。

输出格式（JSON）：
{{
    "intent": "TICKET_CREATE",
    "tasks": [
        {{
            "task_id": "1",
            "task_type": "PROJECTOR_BORROW",
            "reason": "下周二办活动需要使用",
            "metadata": {{
                "start_time": "2024-04-23T14:00:00",
                "end_time": "2024-04-23T18:00:00"
            }}
        }}
    ]
}}

task_type可选值：
- PROJECTOR_BORROW: 投影仪借用
- LAPTOP_BORROW: 笔记本借用
- ROOM_BOOKING: 会议室预订
- IDEA_LICENSE: IDEA激活码申请
- DB_READ_PERMISSION: 数据库只读权限
- DB_WRITE_PERMISSION: 数据库读写权限

用户请求：{user_message}
'''
```

#### 5.3 LangGraph中多任务处理流程
1. 需求理解节点 → 输出tasks列表
2. 遍历tasks，逐个处理：
   - 对于每个task，调用对应的工具
   - 记录tool_result
3. 所有tasks处理完成后，进入回答生成节点
4. 整合所有结果，生成最终回答

#### 5.4 分阶段实现
1. 第一阶段：先实现单任务处理
2. 第二阶段：实现JSON格式输出的任务拆解
3. 第三阶段：实现循环处理多个任务
4. 第四阶段：实现结果整合与汇总回答

## 6. 难点五：模拟审批逻辑

### 难点描述
- 需要模拟负责人审批
- 审批结果需要合理（不能全通过或全拒绝）

### 初步解决方案

#### 6.1 简单随机策略
```java
// 70%通过率，30%拒绝率
public TicketStatus simulateApproval(Ticket ticket) {
    Random random = new Random();
    if (random.nextInt(100) < 70) {
        return TicketStatus.APPROVED;
    } else {
        return TicketStatus.REJECTED;
    }
}
```

#### 6.2 工单状态自动更新
- 创建工单时状态为PENDING
- 可以加一个定时任务（或手动触发）更新状态
- 或者简单点：查询工单时，如果超过1分钟，自动变成APPROVED/REJECTED

## 7. 难点六：两个服务的联调与错误处理

### 难点描述
- Java调用Python，Python调用Java，网络请求可能失败
- 需要处理超时、重试
- 需要统一的错误码和错误信息格式

### 初步解决方案

#### 7.1 统一响应格式
**Java端统一返回格式**:
```java
public class ApiResponse<T> {
    private int code;    // 200成功，500失败
    private String message;
    private T data;
}
```

**Python端统一返回格式**:
```python
class ApiResponse(BaseModel):
    code: int
    message: str
    data: Optional[Any] = None
```

#### 7.2 客户端超时与重试
- Java端调用Agent：设置超时5秒，不重试（对话请求不能重试）
- Python端调用Java：设置超时3秒，失败抛出异常让Agent处理

#### 7.3 本地开发配置
- Java默认端口：8080
- Python默认端口：8001
- 配置文件中可以修改

## 8. 难点七：流式对话（打字机效果）

### 难点描述
- SSE（Server-Sent Events）实现相对复杂
- Java → Agent → 大模型，流式响应需要透传

### 初步解决方案

#### 8.1 第一阶段：先做非流式
- 直接返回完整回复，简单可靠
- 等核心链路跑通了再加流式

#### 8.2 第二阶段：如果时间充裕再加流式
- Java端用SSE
- Python端调用大模型的流式接口
- Java透传Python的流式响应

---

## 9. 难点八：前端开发（最后阶段）

### 难点描述
- React + TypeScript + Ant Design 需要一定前端基础
- 流式响应的前端处理

### 初步解决方案

#### 9.1 简化方案一：用Gradio/Streamlit（推荐）
- 优点：Python写前端，几行代码就能出一个对话界面
- 缺点：不是企业级前端，但作为简历项目足够

#### 9.2 简化方案二：用简单的HTML + JavaScript
- 单页面，不用构建工具
- 用原生EventSource处理SSE

#### 9.3 完整方案：React + Ant Design
- 如果有前端基础，用这个
- 否则选方案一或二

## 10. 总体应对策略

### 10.1 MVP原则（最小可行演示）
- **先跑通，再优化**
- 每个链路先做最简单版本，能演示就行
- 效果不好的地方，用"硬编码"或"模拟"绕过

### 10.2 分阶段验证
每个阶段完成后，必须验证通过再进入下一阶段：
1. 阶段一：基础框架 → 两个服务能启动，能互相调用
2. 阶段二：政策问答 → 能回答"能不能带宠物"
3. 阶段三：资源查询 → 能查"有没有投影仪"
4. 阶段四：审批流 → 能创建工单
5. 阶段五：前端 → 能在页面上对话

### 10.3 备选方案准备
除LangGraph和多任务拆解外，其他难点准备降级方案：

| 难点 | 高级方案 | 降级方案 | 说明 |
|------|---------|---------|------|
| LangGraph | 完整图结构 | ❌ 不降级 | 必须完整实现，核心亮点 |
| RAG | 向量检索 | 关键词匹配 | 向量检索效果不好时用 |
| 多任务拆解 | 自动拆解 | ❌ 不降级 | 必须完整实现，核心亮点 |
| 前端 | React | Gradio/纯HTML | 快速原型可用 |
