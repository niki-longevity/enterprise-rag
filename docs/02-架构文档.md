## 1. 技术选型决策

### 1.1 整体架构模式
- **决策**: 采用 Java API网关 + Python智能服务 的分层架构
- **原因**:
  - Java有成熟的企业级基础设施（鉴权、限流、日志等）
  - Python在AI/LLM生态更丰富（LangGraph、LangChain等）
  - 职责清晰：Java负责业务管控，Agent专注智能逻辑

### 1.2 技术栈详情

| 层级 | 技术选型 | 版本 | 选型原因 |
|------|---------|------|---------|
| 前端 | React + TypeScript + Ant Design | - | 组件丰富，开发效率高，适合简历 |
| API网关/业务层 | Java + Spring Boot | 3.5.10 | 生态成熟，企业级首选 |
| ORM | MyBatis Plus | 3.5.10 | 简化CRUD，适合快速开发 |
| 数据库 | MySQL | 8.0+ | 关系型数据库，适合结构化数据 |
| 智能服务 | Python + FastAPI | 3.11+ | 异步高性能，Python AI生态好 |
| 智能编排 | LangGraph | - | 多步骤Agent编排，状态管理 |
| 向量库 | ChromaDB | - | 轻量本地向量库，开发简单 |
| 大模型 | 国内API（通义千问/豆包） | - | 国内访问稳定，成本可控 |

### 1.3 Git仓库策略
- **决策**: Monorepo（单仓库）
- **原因**:
  - 原子提交：一次commit可包含Java+Python的配套修改
  - 版本统一：两个服务版本天然一致
  - Issue/PR统一：项目管理在一个地方
  - 规模小：适合小项目，没必要多仓库

## 4. 数据模型

### 4.1 资源表 (resource)
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | BIGINT | 主键 | PK, AUTO_INCREMENT |
| name | VARCHAR(100) | 资源名称 | NOT NULL |
| type | VARCHAR(50) | 类型 | ENUM: PROJECTOR, LAPTOP, ROOM, LICENSE |
| status | VARCHAR(20) | 状态 | ENUM: AVAILABLE, IN_USE, MAINTENANCE |
| description | TEXT | 描述 | NULLABLE |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | DATETIME | 更新时间 | DEFAULT CURRENT_TIMESTAMP ON UPDATE |

### 4.2 工单表 (ticket)
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | BIGINT | 主键 | PK, AUTO_INCREMENT |
| ticket_no | VARCHAR(32) | 工单号 | UNIQUE, NOT NULL (格式: TK+年月日+序号) |
| user_id | VARCHAR(50) | 申请人ID | NOT NULL |
| type | VARCHAR(50) | 工单类型 | NOT NULL |
| reason | TEXT | 申请原因 | NULLABLE |
| status | VARCHAR(20) | 状态 | ENUM: PENDING, APPROVED, REJECTED |
| metadata | JSON | 扩展字段 | NULLABLE |
| approved_by | VARCHAR(50) | 审批人 | NULLABLE |
| approved_at | DATETIME | 审批时间 | NULLABLE |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | DATETIME | 更新时间 | DEFAULT CURRENT_TIMESTAMP ON UPDATE |

### 4.3 政策文档表 (policy)
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | BIGINT | 主键 | PK, AUTO_INCREMENT |
| title | VARCHAR(200) | 文档标题 | NOT NULL |
| content | TEXT | 文档内容 | NOT NULL |
| category | VARCHAR(50) | 分类 | ENUM: IT, HR, ADMIN |
| file_path | VARCHAR(500) | 原文件路径 | NULLABLE |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | DATETIME | 更新时间 | DEFAULT CURRENT_TIMESTAMP ON UPDATE |

### 4.4 对话历史表 (chat_history)
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | BIGINT | 主键 | PK, AUTO_INCREMENT |
| session_id | VARCHAR(64) | 会话ID | NOT NULL, INDEX |
| user_id | VARCHAR(50) | 用户ID | NOT NULL |
| role | VARCHAR(20) | 角色 | ENUM: USER, ASSISTANT |
| content | TEXT | 消息内容 | NOT NULL |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |

## 5. API接口定义

### 5.1 前端 → Java接口

#### 5.1.1 发送消息
```
POST /api/chat
Request:
{
  "userId": "user001",
  "message": "我能不能带宠物上班",
  "sessionId": "sess_abc123"  // 可选，用于会话续传
}

Response:
{
  "reply": "根据《员工办公规范》...",
  "sessionId": "sess_abc123",
  "attachments": [
    {
      "type": "POLICY",
      "title": "员工办公规范.md"
    }
  ]
}
```

#### 5.1.2 流式对话（SSE）
```
GET /api/chat/stream?userId=user001&message=...&sessionId=...

SSE Event Stream:
data: {"type": "text", "content": "根"}
data: {"type": "text", "content": "据"}
data: {"type": "text", "content": "《"}
...
data: {"type": "done", "reply": "根据《员工办公规范》..."}
```

#### 5.1.3 查询工单列表
```
GET /api/tickets?userId=user001&status=PENDING

Response:
{
  "tickets": [
    {
      "ticketNo": "TK20240417001",
      "type": "投影仪借用",
      "status": "PENDING",
      "createdAt": "2024-04-17T14:30:00"
    }
  ]
}
```

### 5.2 Java → Agent接口

#### 5.2.1 智能对话
```
POST /api/agent/chat
Request:
{
  "userId": "user001",
  "message": "我能不能带宠物上班",
  "sessionId": "sess_abc123"
}

Response:
{
  "reply": "根据《员工办公规范》...",
  "attachments": [...]
}
```

### 5.3 Agent → Java接口

#### 5.3.1 查询资源
```
GET /api/internal/resources?type=PROJECTOR&status=AVAILABLE

Response:
{
  "resources": [
    {
      "id": 1,
      "name": "投影仪A",
      "status": "AVAILABLE"
    }
  ]
}
```

#### 5.3.2 创建工单
```
POST /api/internal/tickets
Request:
{
  "userId": "user001",
  "type": "PROJECTOR_BORROW",
  "reason": "下周二办活动需要使用",
  "metadata": {
    "startTime": "2024-04-23T14:00:00",
    "endTime": "2024-04-23T18:00:00"
  }
}

Response:
{
  "ticketNo": "TK20240417001",
  "status": "PENDING"
}
```

#### 5.3.3 搜索政策文档
```
GET /api/internal/policies/search?keyword=宠物

Response:
{
  "policies": [
    {
      "id": 1,
      "title": "员工办公规范",
      "content": "3.2 办公区域禁止携带宠物进入..."
    }
  ]
}
```

## 6. 核心逻辑分配原则

### 6.1 必须在Java端的逻辑
- ✅ 数据持久化（所有数据库操作）
- ✅ 事务管理
- ✅ 工单号生成等业务规则
- ✅ 模拟审批逻辑
- ✅ 对话历史存储
- ✅ API鉴权、限流、日志等基础设施

### 6.2 必须在Python端的逻辑
- ✅ 大模型调用
- ✅ LangGraph智能编排
- ✅ RAG检索（向量库操作）
- ✅ 意图识别
- ✅ 任务拆解
- ✅ 工具调用决策

### 6.3 可以在任意端（建议在Java端）
- 政策文档的元数据管理（内容本身给RAG用）
