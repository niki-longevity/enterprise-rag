> **状态**: 开发中

## 1. 技术选型决策

### 1.1 整体架构模式
- **决策**: 采用 Java API网关 + Python智能服务 的分层架构
- **原因**:
  - Java有成熟的企业级基础设施（鉴权、限流、日志等）
  - Python在AI/LLM生态更丰富（LangGraph、LangChain等）
  - 职责清晰：Java负责业务管控，Agent专注智能逻辑

### 1.2 技术栈详情

| 层级 | 技术选型 | 版本 | 选型原因 |
|------|---------|------|---------|
| 前端 | React + TypeScript + Ant Design | - | 组件丰富，开发效率高，适合简历 |
| API网关/业务层 | Java + Spring Boot | 3.5.10 | 生态成熟，企业级首选 |
| ORM | MyBatis Plus | 3.5.10 | 简化CRUD，适合快速开发 |
| 数据库 | MySQL | 8.0+ | 关系型数据库，适合结构化数据 |
| 智能服务 | Python + FastAPI | 3.11+ | 异步高性能，Python AI生态好 |
| 智能编排 | LangGraph | - | 多步骤Agent编排，状态管理 |
| 向量库 | ChromaDB | - | 轻量本地向量库，开发简单 |
| 大模型 | DeepSeek + 阿里云Embedding | - | 国内访问稳定，成本可控 |

### 1.3 Git仓库策略
- **决策**: Monorepo（单仓库）
- **原因**:
  - 原子提交：一次commit可包含Java+Python的配套修改
  - 版本统一：两个服务版本天然一致
  - Issue/PR统一：项目管理在一个地方
  - 规模小：适合小项目，没必要多仓库

## 4. 数据模型

### 4.1 对话历史表 (chat_history)
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | BIGINT | 主键 | PK, AUTO_INCREMENT |
| session_id | VARCHAR(64) | 会话ID | NOT NULL, INDEX |
| user_id | VARCHAR(50) | 用户ID | NOT NULL |
| role | VARCHAR(20) | 角色 | ENUM: USER, ASSISTANT |
| content | TEXT | 消息内容 | NOT NULL |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |

## 5. API接口定义

### 5.1 前端 → Java接口

#### 5.1.1 发送消息
```
POST /api/chat
Request:
{
  "userId": "user001",
  "message": "婚假能请几天？",
  "sessionId": "sess_abc123"  // 可选，用于会话续传
}

Response:
{
  "reply": "根据《休假管理规定》...",
  "sessionId": "sess_abc123",
  "attachments": [
    {
      "type": "POLICY",
      "title": "休假管理规定.md"
    }
  ]
}
```

#### 5.1.2 查询会话历史
```
GET /api/chat/history?sessionId=sess_abc123

Response:
[
  {
    "id": 1,
    "sessionId": "sess_abc123",
    "userId": "user001",
    "role": "USER",
    "content": "婚假能请几天？",
    "createdAt": "2026-04-21T10:00:00"
  },
  ...
]
```

### 5.2 Java → Agent接口

#### 5.2.1 智能对话
```
POST /api/agent/chat
Request:
{
```

```yaml
    username: root
```
