## 核心发现

1. **假设 A 确认成立**：将 top_k 从 5 扩大到 15，复杂问题的 Hit Rate 全面翻倍（+106%~+151%）。目标 chunk 能被检索到，只是排在 6-15 位被 top_k=5 截断了。

2. **Rerank 是正确的解法**：目标 chunk 在 embedding 空间中已经足够接近查询——它们只是被略高排名的相似 chunk 挤出了前 5。一个精排模型可以轻松纠正这个顺序。

3. **多Query + top_k=15 是最强组合**：复杂问题 Hit Rate 0.739，33.3% 完全命中，0% 完全未命中。这是唯一让复杂问题达到可用水平的方案。

4. **简单/口语化问题接近天花板**：top_k=15 + 直接改写，简单问题 97.5%、口语化 100%。继续优化空间不大。

5. **HyDE 是单查询策略中最优的**：复杂问题 Hit Rate 0.606、0% 未命中。不需要多次查询就能覆盖所有复杂问题。

## 实施方案：Rerank 精排

### 架构

```
用户Query → 向量初检(top_k=15) → Reranker精排 → 取前5返回
```

### 实现

LlamaIndex 内建 `SentenceTransformerRerank` node postprocessor，中文用 BGE-reranker-v2-m3：

```python
# retriever.py 改动
from llama_index.core.postprocessor import SentenceTransformerRerank

reranker = SentenceTransformerRerank(
    model="BAAI/bge-reranker-v2-m3",
    top_n=5
)

def search(query: str, top_k: int = 3):
    retriever = index.as_retriever(similarity_top_k=15)
    nodes = retriever.retrieve(query)
    nodes = reranker.postprocess_nodes(nodes, query_str=query)
    return [{"content": n.text, "metadata": n.metadata} for n in nodes[:top_k]]
```

### 成本

- 改动量：低（retriever.py 加 ~15 行，requirements.txt 加 1 个依赖）
- 额外延迟：~100-200ms（本地部署 reranker model）
- 额外模型：BGE-reranker-v2-m3（~1.5GB，首次下载后本地加载）

### 预期收益（基于 top_k=15 实测数据推算）

| 场景 | 当前 k=5 | Rerank 后 | 提升 |
|------|---------|-----------|------|
| 简单问题 | 0.875 | ~0.90 | +3% |
| 复杂问题 | 0.228 | **~0.45-0.50** | **+100%** |
| 口语化 | 0.776 | ~0.90 | +16% |

> 注：Rerank 不可能 100% 还原 top_k=15 的效果（精排模型有精度损失），但 Hit Rate 翻倍是合理的预期。

### Rerank 后再叠加多Query（后续优化）

如果 rerank 后复杂问题仍不满足需求，加多Query 扩展：

```
用户Query → 多Query扩展 → 每个子Query: 初检(k=10) → 合并去重 → Rerank → 取前5
```

多Query(k=15) 实测 0.739，加上 Rerank 后有望接近 **0.7-0.8**。

## 其他方案（后续考虑）

### 按章节语义切割

政策文档的 Markdown 层级清晰，按 `##` 小节切割能提升 chunk 语义完整性。但 top_k=15 验证已证明 embedding 质量够用，语义切割作为增量优化排在 rerank 之后。

### 混合检索（BM25 + 向量）

对于"D级"、"300元"等精确术语查询有帮助。当前 rerank 已经能解决大部分排名问题，混合检索作为进一步补强。
